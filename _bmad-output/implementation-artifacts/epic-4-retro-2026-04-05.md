# Epic 4 Retrospective — Two-Way Sync with Conflict Safety
Date: 2026-04-05

## Outcome

**3/3 stories done (100%).** Conflict detection, delta detection + two-way sync engine, sync command — all shipped.

---

## What Went Well

- **Conflict copy naming is well-designed.** `filename.conflict-YYYY-MM-DD` suffix-after-extension pattern is unambiguous and produces no false matches during detection.
- **StateDB upsert pattern is solid.** `INSERT OR REPLACE` keyed on `local_path` with full sync record ensures idempotent state transitions. Closing in `finally` block applied consistently across all commands.
- **Sync command exit code handling is correct (after fix).** Human mode and JSON mode both exit 1 on partial errors after the review patch in 4-3.

## What Didn't Go Well

- **Clock mismatch in `handleRemoteOnlyFile`.** `stored.lastSyncMtime === remoteItem.mtime` compared a post-download local filesystem mtime against a remote server mtime — two different clocks that will never match. Every remote file re-downloaded on every run. Caught in review (4-2). A fundamental misunderstanding of what "last sync mtime" represents.
- **Timezone bug in `buildConflictCopyName`.** Used local-time date fields (`getFullYear/getMonth/getDate`) while `sync-engine.ts` used UTC (`toISOString().slice(0,10)`) for the remote conflict path. Near-midnight in non-UTC timezones, the local and remote conflict copies get different date suffixes, corrupting the StateDB record. Caught in review (4-1).
- **JSON mode exit code inconsistency (4-3).** When `result.errors` was non-empty, JSON mode called `formatSuccess` and exited 0. Human mode correctly exited 1. This would silently break CI pipelines using `--json`. Caught in review.

## Key Lessons

1. **Mtime comparisons: always compare like-for-like clocks.** Remote mtime from SDK ≠ local filesystem mtime after download. Use stored hash as the primary change detector; mtime as a hint only.
2. **Date serialization: commit to UTC everywhere.** Local-time date components in any serialized string is a timezone bug waiting to happen. `toISOString().slice(0,10)` is the correct pattern throughout.
3. **JSON mode exit codes must match human mode exit codes.** Review should always verify both output paths for commands with error conditions.
