# Epic 7 Retrospective — End-to-End Test Suite
Date: 2026-04-05 (updated — all 3 stories complete)

## Outcome

**3/3 stories done (100%).** Binary e2e tests (7-1), live integration tests (7-2), and E2E CI workflow (7-3) all shipped.

---

## What Went Well

- **E2e test isolation is solid (7-1).** `XDG_DATA_HOME` override per test, binary pre-flight skip when binary absent, `spawnSync` + stdout/stderr assertion pattern — clean and reproducible.
- **SDK initialization worked (7-2).** `createLiveDriveClient` factory with `buildHttpClient`, `buildAccount` (SRP key derivation), `OpenPGPCryptoWithCryptoProxy` wired up. The full encrypted upload→download cycle works end-to-end.
- **Code review process caught a critical AC4 bug before it could be missed (7-2).** The `controlledClient` object spread issue (prototype methods not copied → uploadFile undefined → TypeError in SyncEngine → conflicts never recorded) would have been impossible to catch without a real credentials test run. The review layer caught it.
- **CI workflow story was clean and fast (7-3).** Three parallel review agents produced 20 raw findings; 17 dismissed. The architect consultation produced a clean P1/P3-implement, P2-defer split — no argument about it. 2 patches applied, 12 tests passing.
- **YAML structure testing with js-yaml is effective.** Using `js-yaml` to parse and assert on workflow YAML structure (triggers, step ordering, secrets wiring, SHA pinning, build target) is lightweight and caught real coverage gaps the initial implementation missed.

## What Didn't Go Well

- **Story 7-2 was the most complex story in the project.** Implementing the real SDK initialization required: an OpenPGP crypto proxy adapter (`openpgp-proxy.ts`), a Proton account service (`account-service.ts`), an SRP module adapter (`srp-module.ts`), and full implementations of `uploadFile`, `downloadFile`, `listFolder`, `getFileMetadata`. This complexity was the direct consequence of the 4-story deferral from Epic 3. The accumulated stub debt was large.
- **Object spread on class instance (controlledClient) is a JavaScript footgun (7-2).** `{ ...driveClient, listFolder: override }` copies only own enumerable properties — prototype methods are absent on the resulting object. This is a subtle JS semantics issue that TypeScript's type system does not prevent (cast hid it). Lesson: never spread class instances to create mocks — use `Object.create()`.
- **`/integration-tests/` parent folder accumulated across test runs (7-2).** A minor AC5 violation — cleaned up by changing to a flat remote path in the code review fix.
- **Pre-existing `cli.test.ts` binary compilation failure** (`spawnSync("bun", ...)` with bun not in PATH) has carried through all epics unresolved. Not introduced by Epic 7 but now the only failing test.
- **P2 bun-version test was correctly deferred but reveals a structural gap.** There is no single source of truth for the Bun version across ci.yml, release.yml, and e2e.yml. Architect consultation surfaced this explicitly — the right fix is a `.tool-versions` or `engines` field, not a brittle test that breaks on every Bun upgrade.

## Key Lessons

1. **Deferred SDK stubs accumulate compound debt.** A 4-story deferral from Epic 3 became the single largest implementation effort in the project. Future stub deferrals should have a hard deadline story in the same epic, not punted to a later one.
2. **Never spread class instances to create test doubles.** `{ ...instance }` omits prototype methods. Use `Object.create(instance)` with method overrides, or a proper wrapper class.
3. **Integration tests without real credentials must be verified by someone who has them.** The `test.skipIf(SKIP)` pattern is correct for CI, but it means code quality of integration tests depends entirely on human + review-layer scrutiny. There is no automated validation loop.
4. **Architect consultation on review patches is cheap and effective.** The three-agent parallel review + architect triage produced a sharper decision on P2 than any single reviewer would have reached. The "source-of-truth needed first" framing is a reusable pattern.
5. **YAML structure tests are a valid and lightweight quality gate for CI configuration.** A workflow file is infrastructure code; testing it deserves the same red-green discipline as application code.
