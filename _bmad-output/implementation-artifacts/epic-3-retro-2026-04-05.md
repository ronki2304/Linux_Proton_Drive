# Epic 3 Retrospective — Manual File Operations
Date: 2026-04-05

## Outcome

**3/3 stories done (100%).** DriveClient SDK wrapper with retry policy, upload command, download command — all shipped. SDK methods were stubs at this stage (real implementation deferred to Epic 7).

---

## What Went Well

- **SDK boundary held cleanly.** `src/sdk/client.ts` as the sole `@protontech/drive-sdk` import point was respected throughout. No accidental cross-boundary imports.
- **Retry policy is robust.** `withRetry()` with 3 attempts, 1s/2s/4s backoff, non-retryable error short-circuit (`AuthError`, `ConfigError`, `ValidationError`) implemented once and reused across all SDK methods.
- **Atomic download pattern established.** Write to `.dl-tmp-<timestamp>-<random>` then `renameSync` — prevents corrupt partial files at destination. Used consistently.

## What Didn't Go Well

- **File system race condition in upload (TOCTOU).** `collectFiles` used `statSync` after `readdirSync` — file deleted between the two calls throws uncaught ENOENT, bypassing `formatError` and exposing a raw OS error. Caught in review (3-2).
- **Download flattens subdirectory structure.** `path.basename(item.remotePath)` in directory download discards subdirectory paths — files with identical basenames silently overwrite each other. Caught in review (3-3). Fundamental path handling oversight.
- **SDK methods remained stubs.** `uploadFile`, `downloadFile`, `listFolder`, `getFileMetadata` all returned empty/threw NOT_IMPLEMENTED. This was intentional (deferred to 7-2) but meant 3 full stories shipped functionality that couldn't make real API calls. The integration test suite (7-2) was the actual validation gate.

## Key Lessons

1. **Directory traversal → always use `try/catch` around `statSync`.** File system state between `readdirSync` and `statSync` is not stable. Race conditions are real on busy filesystems.
2. **Remote path → local path mapping must preserve directory structure.** Using `path.basename` alone is only valid for single-file downloads. Directory downloads need path suffix preservation.
3. **Stub-first is a valid approach but needs explicit integration test coverage planned.** The SDK stubs carried real risk of divergence from the actual SDK API — that risk materialized as the largest single story in the project (7-2).
