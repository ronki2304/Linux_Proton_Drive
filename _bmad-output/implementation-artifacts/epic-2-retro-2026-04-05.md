# Epic 2 Retrospective — Secure Authentication
Date: 2026-04-05

## Outcome

**4/4 stories done (100%).** SRP-6a auth, credential store with OS keychain + headless fallback, auth commands, TOTP 2FA — all shipped.

---

## What Went Well

- **SRP implementation is correct and complete.** Custom SRP-6a variant with bcrypt v4 password expansion implemented from scratch. All cryptographic edge cases (u=0, B mod N=0, server proof verification) explicitly handled.
- **Credential store abstraction is clean.** `CredentialStore` interface with `KeyringStore` / `FileStore` implementations — swappable without touching commands. Mock boundary at interface level held throughout all test suites.
- **2FA architecture is non-blocking.** `TwoFactorRequiredError` carrying the partial session token cleanly decouples the auth flow from TOTP handling. Interactive TTY path and headless TOTP_NO_TTY path both handled without polling or hanging.

## What Didn't Go Well

- **Terminal I/O was a persistent pain point.** Story 2-3 required rewriting `promptPassword` (readline interface conflicted with raw stdin listener). Story 2-4 hit CRLF terminal input corrupting TOTP code (`"123456\r"` failing `/^\d{6}$/`). Two separate terminal I/O bugs — neither caught in initial implementation.
- **Security edge case missed in FileStore.** Path traversal via unsanitized key parameter in `FileStore` caught in review (2-2). The fix (`validateKey()` with strict regex) was clean, but shouldn't have been missed — `FileStore` writes to disk using the key as a filename component.
- **fetchJson 422 handling is a known gap.** HTTP 422 (Proton API application errors) bypasses `!response.ok` and is parsed as `T`. Deferred — intentional for now but surfaces as AUTH_FAILED in edge cases.

## Key Lessons

1. **Terminal I/O requires explicit CRLF and raw-mode testing.** Any stdin listener must strip `\r` before matching patterns. readline and raw data listeners are incompatible — pick one.
2. **Filename-as-parameter → always validate.** Any function that uses caller-supplied input to construct a file path needs a strict allowlist, not a blocklist.
3. **Proton API uses 422 for application errors.** Document this; callers see AUTH_FAILED instead of a useful code.
