## Epic Summary

- **Epic:** 1 — App Foundation & Authentication
- **Stories:** 13/13 completed (100%)
- **Test Coverage:** 110 UI tests (pytest) + 16 engine tests (node:test) = 126 total
- **Code Review Patches:** 49 fixes applied across Stories 1.1–1.9
- **Deferred Items:** 35 total (9 triaged to Epic 2 Story 0, rest stays with planned epics)

## Successes

1. **100% story completion** — clean sweep on first epic, all 13 stories delivered
2. **Intense PRD/architecture/UX prep work paid off massively** — no design debates during implementation, crystal-clear acceptance criteria on every story, the planning artifacts are the project's competitive advantage
3. **Strong architectural boundaries established early** — `sdk.ts` boundary with CI enforcement script, Blueprint rule for all widget structure, `errors.py`/`errors.ts` as zero-import leaf modules, IPC protocol with 4-byte length-prefix + JSON
4. **Code reviews caught 49 real bugs** in Stories 1.1–1.9 — race conditions, TOCTOU vulnerabilities, security issues, missing error guards
5. **Test suite grew steadily** — from 0 to 126 tests, every story from 1.3 onward added tests with regression verification
6. **project-context.md** — 89 rules encoding all gotchas discovered, saves re-learning time per story

## Challenges

1. **Code review dropped off for Stories 1.10–1.13** — stories were marked `done` before Jeremy's review; findings were not captured in story files. Jeremy had to force reviews to happen.
2. **Git discipline missing** — 5 commits for 13 stories, multiple stories batched into single commits, no granular revert capability. Project conventions specify `feat/<story-id>-short-desc` branches but none were used.
3. **35 deferred items accumulated** — 9 directly relevant to Epic 2 stability, including IPC backpressure, restart reentrancy, auth failure paths, and test fixture pollution
4. **No real UI test yet** — all 110 "UI tests" are unit tests with mocked GTK objects. No pixels-on-screen verification. First real UI test comes with Epic 2 Story 2.10 (Flatpak Build Validation).
5. **`sys.modules` pollution in test fixtures** — module-level GI mocks leak across test sessions, pattern growing harder to fix with each story added

## Key Insights

1. **Planning artifact quality directly correlates with implementation speed** — the PRD, architecture, and UX spec were intense to produce but eliminated ambiguity during implementation
2. **Solo dev + AI agents needs an adapted workflow** — the human must be the quality gate; dev agent should never self-certify its own work
3. **Technical debt compounds at epic boundaries** — addressing it in Story 0 before building new features prevents cascading issues
4. **First real UI test is a milestone** — unit tests are necessary but not sufficient; pixels-on-screen validation is still ahead

## Epic 2 Story 0 — Technical Debt Items

| # | Item | Source | Why it matters for Epic 2 |
|---|------|--------|--------------------------|
| 1 | `writeMessage` ignores `socket.write()` backpressure | 1-3 W3 | Continuous sync progress events will flood the socket |
| 2 | `restart()` reentrancy risk if error callback triggers restart | 1-5 W6 | Sync failures in Epic 2 will hit this path |
| 3 | `_write_message` failure doesn't tear down stale connection | 1-5 W4 | Sync disconnects need clean teardown |
| 4 | `cleanup()` doesn't close connection explicitly | 1-5 W1 | Engine lifecycle must be solid for long-running sync |
| 5 | `send_command` queues dict by reference — mutation risk | 1-5 W5 | Sync queue replay could mutate queued commands |
| 6 | Malformed JSON messages silently dropped with no logging | 1-4 W2 | Sync debugging impossible without visibility |
| 7 | `_on_auth_completed` transitions to main UI even if token storage fails | 1-8, 1-6 | Setup wizard (Story 2.4) reuses this flow |
| 8 | WebView network session not explicitly cleared after auth | 1-9 W4 | Auth reuse in setup wizard could leak sessions |
| 9 | `sys.modules` pollution in test fixtures | 1-4 W6, 1-8 | 110+ tests and growing — fix now or never |
