## Challenges

1. **29 pre-existing auth test failures — unsustainable** — present since Story 3-0b, carried unchanged through all 6 stories. Tests in `test_auth_completion.py`, `test_auth_window.py`, `test_credential_store.py`, `test_main_routing.py` — the auth path. Jeremy called it explicitly: tests that fail "normally" train everyone to ignore red. Captured as critical prep item before Epic 4.
2. **Resource lifecycle gaps** — Story 3-1 review found 4 patches: in-flight `checkFn` emitting after `stop()`, socket not destroyed on error, `networkMonitor` not stopped on SIGTERM, `on_online()` calling `update_all_synced()` without syncing check. Recurring blindspot: "if you open it, close it" not consistently applied in initial implementations.
3. **Non-atomic compound DB operations** — Story 3-3: `upsertSyncState` + `dequeue` were two separate writes. Process crash between them = remote uploaded, queue entry never dequeued. Caught in review; fixed via `commitUpload`/`commitTrash`/`commitDequeue` `db.transaction()` wrappers. Should have been there from the start.
4. **SDK pre-release instability** — `fetchAndDecryptAllKeys` API changed during 3-0a, required rewriting `makeMockAdapter` in `sdk.test.ts`. Third SDK-related surprise in this project. `sdk.ts` boundary absorbed it in one file.
5. **11 review patches in Story 3-3** — highest review patch count of any story. Complex story, but also reflects that resource lifecycle and atomicity issues weren't caught before review. See action items 2 and 3.

---

## Key Insights

1. **The "pre-existing" label is a debt trap** — once something is labelled pre-existing, it defers forever. The 29 auth failures are the proof. Jeremy broke the pattern by naming it unsustainable. Action: anything labelled "pre-existing" in a story review gets a deadline, not a deferral.
2. **Story 0 is load-bearing** — the pattern of addressing debt before feature work is what makes 100% retro follow-through possible. Without Story 0, action items compete with feature stories and lose.
3. **Proactive refactoring in Story 3-3 paid forward to Story 3-4** — the `_set_dot_state` matrix rewrite, the sequential per-entry shape, the `busy` enum design. These weren't required for 3-3's ACs but they made 3-4 and future 2-12 work cleaner. Worth naming explicitly so the team continues this pattern.
4. **Documented smoke tests improve review quality** — stories with explicit smoke test notes in the Dev Agent Record (3-0a, 3-3) had cleaner reviews. Stories without it required reviewers to assume worst-case. Add smoke test documentation as an explicit expectation.

## Significant Discoveries

### 1. Story 2-12 Architecture Gap (Critical)

**Discovery:** Jeremy's insight during Story 3-3 review — "Why don't you put file updates in the same queue instead of a new process?" — identified that `startSyncAll` (tree-walk-driven) and `replayQueue` (queue-driven) race in concurrency edge cases, duplicate conflict-detection logic, and double the test surface.

**Resolution:** Story 2-12 documented and activated before Epic 4. Story 3-3's `replayQueue()` was intentionally shaped as the seed of the future `drainQueue()`. Jeremy confirmed: 2-12 is a must-have before Epic 4 starts.

### 2. Proton SDK Team Response (Informational — Plan Unchanged)

**Discovery:** Proton team replied to GitHub issue: "working on a solution to provide the SDK with everything needed." Provided web client monorepo reference.

**Resolution:** Plan unchanged — SDK has been pending 4 years; we ship on what exists. Web client reference is a bounded spike for opportunistic pattern extraction, not a roadmap dependency. `sdk.ts` boundary absorbs any future SDK changes in one file.
