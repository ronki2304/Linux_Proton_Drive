## Challenges

1. **Stale Flatpak binary caused debugging confusion** — the installed binary was compiled from code weeks older than what was on disk. Logs showed `workList: 0 items` from a ghost binary. No warning, no version mismatch signal. Consumed the first half of this retro session.
2. **Deferred items accumulating without triage** — 100+ items across 4 epics in `deferred-work.md`. The SM reads retro action items, not the deferred file. Items pile up; none are closed without explicit effort. Jeremy read the full list and found it unmanageable.
3. **External research not captured** — Action Item 4 (Proton web client research) was completed by Jeremy via GitHub issue + team reply, but this wasn't written into any story file. Future dev agents had no visibility into the validation outcome.
4. **`meson test` blasting developer CPU** — dev agent runs `meson test -C builddir` (including in background) believing it to be idle. It is not idle. It saturates the host CPU and freezes the machine until killed. Required Jeremy to manually terminate the process.

---

## Key Insights

1. **Deferred items are an archive, not a living list** — the file grows every epic but never shrinks. Items from Story 1-1 are still open. The fix is at retro time: every new deferred item from the completed epic gets a verdict out loud before action items are synthesized.
2. **External work must land in the codebase** — if a dev agent can't read it, it doesn't exist. Research, validations, and external team feedback belong in `project-context.md`, not in GitHub threads.
3. **`meson test` ≠ `meson compile`** — compile is fast and acceptable; test saturates CPU. The project-context.md rule must be hard and specific. Done during this retro.
4. **Live testing is part of the retro** — Jeremy's mid-session live test found a real bug. The diagnostic loop (add logging → rebuild → install → test → diagnose) took 30 minutes. This is normal and healthy. Build the habit.
