# Epic 1 Retrospective — Foundation: Validated Project Skeleton
Date: 2026-04-05

## Outcome

**5/5 stories done (100%).** Clean foundation. All subsequent epics built on this without revisiting core decisions.

---

## What Went Well

- **SDK compatibility spike (1-1) paid off immediately.** `@napi-rs/keyring` bundled cleanly — no `dbus-next` fallback needed. Discovering this on day one avoided a parallel code path across the entire project.
- **Architecture boundaries held for all 7 epics.** `src/sdk/client.ts` as sole SDK import point, one-way dependency rule, XDG paths — none of these needed revisiting.
- **Dev execution was smooth.** Well-specified stories with clear AC boundaries → straightforward implementation, no mid-story redirects.
- **JSON output contract established early (1-3).** Every subsequent command had a clear spec to implement against.

## What Didn't Go Well

- **Error handling consistently shallow on first pass.** All 5 code reviews flagged error path gaps: string-matching on status codes (1-1), `parseAsync` non-Error rejection (1-2), `console.error` in catch block (1-3), missing YAML array guard (1-4), uncaught StateDB.init errors (1-5). Root cause: ACs specified happy paths precisely but error scenarios lightly.
- **Bun toolchain surprises added friction.** `bunfig.toml` rejecting empty array, `createRequire` not resolving in compiled binaries — both required wall-hitting to discover. Not preventable, but worth documenting for future Bun version upgrades.
- **DriveClient stub deferred without clear plan.** `sdkClient = null` placeholder was correct for the spike but the deferral note lacked a specific target story. It was ultimately resolved in story 7-2 — much later than ideal.

## Key Lessons

1. **Write at least one error-path AC per story** — happy path correctness is table stakes; error path coverage needs explicit specification.
2. **Bun toolchain gotchas live in project-context.md** — the `import pkg with { type: "json" }` and `bunfig.toml` findings were captured there for future agents.
3. **Architecture spikes are worth front-loading** — the risk-first approach of 1-1 saved the project from a potentially blocking toolchain issue.
